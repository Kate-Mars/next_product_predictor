## Формулировка задания<a id="form"></a>
Предоставляется синтетический датасет e-commerce с пользовательскими сессиями просмотра товаров.
Каждая сессия — это последовательность ID товаров, которые один пользователь просмотрел по порядку.
**Каждая сессия содержит не менее 3 товаров.**

Данные хранятся в формате JSON Lines (_sessions.jsonl_): каждая строка — это JSON-массив, представляющий одну сессию.
Элементы массива — целочисленные ID товаров:
```
[17, 42, 8, 103, 42]
[5, 99, 203, 17]
[203, 17, 88, 42, 7, 19]
[11, 11, 42]
```
Товары в сессии упорядочены по времени просмотра: первый элемент — самое раннее взаимодействие, последний — самое позднее. \
Загрузить данные можно стандартными средствами Python:
```
import json

sessions = []
with open("sessions.jsonl") as f:
    for line in f:
        line = line.strip()
        if line:
            sessions.append(json.loads(line))

# sessions — список списков целых чисел
print(f"Всего сессий: {len(sessions)}")
print(f"Первая сессия: {sessions[0]}")
```
Мною был реализован проект рекомендации следующего элемента на основе данных пользовательских сессий

## Шаг 2<a id="step-2"></a>
После того как я проанализировала данные, нужно было правильно разбить их на обучающую и тестовую выборки. В задании требовалось использовать определённую схему.
```
Для сессии [380, 293, 262, 114, 123, 345, 335, 245, 272, 293, 233, 302, 247, 290, 260, 341, 293]:

• Обучающая часть: [380, 293, 262, 114, 123, 345, 335, 245, 272, 293, 233, 302, 247, 290, 260, 341]
• Тестовая цель: 293
```
### Почему случайное разбиение здесь не подходит?<a id="why"></a>
Если бы я использовала случайное разбиение (случайно взяла тестовым элементом например 2 или 3 и тд.), то модель могла бы «заглядывать в будущее». Когда пользователь просматривает товары, мы в каждый момент времени знаем только то, что он уже посмотрел, и не знаем, что он посмотрит дальше. Случайное разбиение нарушило бы эту временную структуру: тестовый товар мог бы случайно оказаться из середины сессии, а следующие за ним товары — в обучающей выборке. Тогда модель получила бы информацию, которая на самом деле недоступна в момент предсказания, и оценка качества получилась бы нереалистичной.

## Шаг 3<a id="step-3"></a>
На этом шаге я построила **граф переходов** между товарами на основе обучающих данных. Нужно для каждого товара i понять, с какой вероятностью следующим будет просмотрен товар j - найти условную вероятность P(j | i). \
Я прошлась по всем обучающим сессиям и для каждой пары соседних товаров (a, b) увеличивала счётчик перехода от a к b. Параллельно я считала общее количество исходящих переходов для каждого товара a. Затем для каждого товара a и всех его соседей b я вычислила вероятность перехода как отношение количества переходов (a, b) к общему числу переходов из a.
```
Граф переходов:
ID товара -> ID товара (вероятность)
  54 -> 53 (0.169), 329 (0.087), 335 (0.071)
 335 -> 54 (0.111), 373 (0.087), 245 (0.053)
  53 -> 54 (0.244), 65 (0.070), 335 (0.067)
 114 -> 293 (0.108), 380 (0.102), 54 (0.089)
 260 -> 149 (0.105), 92 (0.078), 54 (0.078)
 293 -> 114 (0.314), 380 (0.082), 54 (0.082)
...
```
### Обработка редких и отсутствующих переходов
1. **Переход существует, но его вероятность очень мала** \
 В выводе графа я использовала порог alpha = 0.001. Если вероятность перехода меньше этого порога, в отладочном выводе я помечала такой переход как «маловероятный», чтобы визуально отделить значимые переходы.
2. **Товар ни разу не встречался при обучении** \
Такого не случилось, так как все 400 товаров встретились в обучающей выборке хотя бы один раз. В последующем нам нужно будет, чтобы у графа было минимум 10 переходов, добьем их самыми популярными элементами.

## Шаг 4<a id="step-4"></a>
На этом шаге я построила модель, которая для каждой сессии предсказывает топ-10 наиболее вероятных следующих товаров на основе последнего просмотренного товара.
### Первая попытка - более сложная модель (`recommend`)<a id="m1"></a>
Сначала я решила, что если модель будет учитывать не только последний товар, а всю историю сессии, она сможет давать более точные и персонализированные рекомендации. Я построила функцию `recommend`, которая работала так:
- Каждый товар из истории сессии давал веса своим «соседям» по графу переходов.
- Чем дальше товар отстоит от конца сессии, тем меньше его вклад — я использовала экспоненциальное затухание с коэффициентом _decay = 0.75_.
- Я дополнительно собрала статистику по цепочкам из трёх товаров. Если в истории встречалась пара (A, B), я смотрела, какие товары C чаще всего шли после неё, и добавляла их в кандидаты с повышенным весом (_trip_w = 1.8_ против _pair_w = 1.0_ для обычных пар).
- Все товары, которые уже встретились в текущей сессии, удалялись из списка кандидатов(всего 14% что они повторятся). \

Я ожидала, что такая модель будет лучше улавливать контекст. Однако после тестирования я увидела, что эта модель угадывает следующий товар только в `33.7%` случаев. Это хуже, чем даже простой бейзлайн из десяти самых популярных товаров. Я думаю, причина в том, что _тройки переходов_ встречаются значительно реже пар. Исключение уже просмотренных товаров могло отсекать кандидатов — примерно 14% просмотров являются возвратами к ранее увиденным товарам.

### Вторая попытка - простая модель (`recommend_model`)<a id="m2"></a>
Я использую граф переходов, построенный на предыдущем шаге. Для каждой тестовой сессии я беру её последний товар last и смотрю в графе, какие товары чаще всего шли после него. Ранжирую их по убыванию вероятности перехода P(j | last) и возвращаю первые десять. Если кандидатов меньше k, я дополняю список самыми популярными товарами (из ранее собранного списка `top_items`)(говорили об этом в анализе :D), исключая те, которые уже есть в рекомендациях, и сам товар last - это гарантирует, что модель всегда возвращает _ровно 10 товаров_.
Модель, опирающаяся только на последний товар, угадала следующий товар в 1319 случаях из 2565. (`51.4%`)

## Шаг 5<a id="step-5"></a>
На этом шаге я оценила качество модели по метрике `Hit@10` и сравнила её с простым бейзлайном
Hit@10 - метрика, которая показывает, в какой доле случаев истинный следующий товар попадает в топ-10 предсказанных моделью товаров
Бейзлайн - всегда рекомендовать 10 самых популярных товаров (из списка top_items, составленного по обучающей выборке), независимо от истории пользователя. Это минимальный порог, который любая модель должна превзойти, чтобы считаться полезной
Я прогнала все три подхода через `hit_at_k`:
```
Модель: 51.42 %
Неэффективная модель: 33.72 %
Бейзлайн: 38.40 %
```
Моя итоговая модель, основанная на графе переходов и учитывающая последний просмотренный товар, показала результат `51.42%`. Это более чем в половине случаев правильно угадывает следующий товар среди десяти кандидатов. Альтернативная модель с учётом всей истории сессии, троек и затухания показала лишь `33.72%` — даже хуже бейзлайна. Это подтвердило моё решение отказаться от неё в пользу более простого подхода.
